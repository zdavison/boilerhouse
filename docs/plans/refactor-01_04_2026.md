## Bug Fixes

### B1. Pool not replenished when instance is manually destroyed

**Files:** `apps/api/src/routes/instances.ts:161-186`, `apps/api/src/bootstrap.ts:253-260`

**Problem:** When a user manually destroys a pool instance via `POST /instances/:id/destroy`, the pool is not replenished. The destroy route calls `instanceManager.destroy()` which clears the `poolStatus` marker and emits an `instance.state: destroyed` event via the audit logger, but nothing listens for that event to trigger replenishment. Pool replenishment only happens today during `TenantManager.release()` and `PoolManager.prime()`.

**Root cause:** The destroyâ†’replenish link only exists inside `TenantManager.release()` as a direct call (`this.poolManager.replenish(...).catch(() => {})`). Manual destroys bypass TenantManager entirely.

**Fix:** Add an EventBus listener in `bootstrap.ts` (alongside the existing `ResourceLimiter` listener at line 253) that calls `poolManager.replenish(workloadId)` on any `instance.state: destroyed` event where `workloadId` is present:

```typescript
eventBus.on((event) => {
  if (event.type === "instance.state" && event.status === "destroyed" && event.workloadId) {
    poolManager.replenish(event.workloadId).catch((err) => {
      log.warn({ workloadId: event.workloadId, err }, "Pool replenish after destroy failed");
    });
  }
});
```

This is safe because `replenish()` is already a no-op when the pool is at target size. It also fixes any other edge case where a pool instance is destroyed outside the normal release flow (e.g., recovery, drain race conditions).

**Scope:** `bootstrap.ts` only â€” ~5 lines. Add a unit test that destroys a pool instance via the API and verifies replenishment is triggered.

**Note:** This makes the direct `poolManager.replenish()` call in `TenantManager.release()` redundant. It can be removed as part of fix #9 (ClaimOrchestrator extraction) to avoid double-replenish, though the duplication is harmless since replenish is idempotent.

## Phase A â€” Critical Fixes

### 1. Optimistic concurrency in `applyTransition`

**File:** `apps/api/src/transitions.ts:50-53`

**Problem:** `applyTransition` accepts `currentStatus` as a parameter but writes the new status without verifying the row still has that status. Any `await` between reading and writing creates a race window.

**Fix:** Change the `db.update()` to include `AND status = ?` in the WHERE clause. If zero rows are updated, throw a conflict error. This is a one-line change in the generic `applyTransition` function:

```typescript
const result = db.update(table)
  .set({ status: next })
  .where(and(eq(idColumn, id), eq(table.status, currentStatus)))
  .run();
if (result.changes === 0) {
  throw new InvalidTransitionError(/* entity */, currentStatus, event, "status changed concurrently");
}
```

**Scope:** `transitions.ts` only. All callers already pass `currentStatus`.

### 2. Decompose `bootstrap()` into a service container

**File:** `apps/api/src/bootstrap.ts:134-370`

**Problem:** 290-line god function creating ~15 objects with complex interdependencies. Hard to test, no shutdown lifecycle.

**Fix:**
- Extract service groups into composable factory functions:
  - `createStorageServices(config)` â†’ `{ overlayStore, encryptedOverlayStore }`
  - `createRuntimeServices(config, db, audit, nodeId)` â†’ `{ runtime, instanceManager, secretStore }`
  - `createMonitoringServices(config, eventBus, ...)` â†’ `{ meter, tracer, containerStatsPoller }`
  - `createTenantServices(...)` â†’ `{ tenantManager, tenantDataStore, idleMonitor, poolManager }`
- Keep `bootstrap()` as the top-level orchestrator that calls these factories
- Return a `shutdown()` handle from `bootstrap()` (see fix #3)

**Scope:** `bootstrap.ts`, new file `apps/api/src/service-groups.ts`. No changes to individual service classes.

### 3. Graceful shutdown

**Problem:** `WorkloadWatcher`, `ContainerStatsPoller`, `IdleMonitor`, and `WatchDirsPoller` are started in `bootstrap()` but never stopped. On SIGTERM, in-flight releases could lose overlay data.

**Fix:**
- Add `stop()` methods to `WorkloadWatcher`, `ContainerStatsPoller`, `IdleMonitor`, `WatchDirsPoller` (most already have them or are trivial to add â€” clear interval timers)
- Add `AppContext.shutdown()` that:
  1. Stops accepting new HTTP connections
  2. Waits for in-flight claims/releases (drain `inflightClaims` map)
  3. Stops pollers and watchers
  4. Drains pools
  5. Closes DB
- Wire `process.on("SIGTERM", ...)` and `process.on("SIGINT", ...)` to `shutdown()` in the CLI entrypoint

**Scope:** `bootstrap.ts`, CLI entrypoint, minor additions to poller/watcher classes.

### 4. Retry overlay extraction before destroying on release failure

**File:** `apps/api/src/tenant-manager.ts:119-130`

**Problem:** Overlay extraction failure during release destroys the instance, permanently losing the tenant's session data. Only the previous snapshot survives.

**Fix:**
- Retry overlay extraction with exponential backoff (e.g., 3 attempts: 1s, 2s, 4s)
- If all retries fail, transition instance to a `release_failed` status instead of destroying
- Emit a `tenant.release_failed` event via EventBus for metrics/alerting
- Add a new API endpoint or admin action to retry failed releases or force-destroy
- Add `release_failed` to the instance state machine as a terminal-ish state that allows `destroy` transition

**Scope:** `tenant-manager.ts`, `packages/core/src/state-machine.ts` (new state), new route for retry/force-destroy.

### 7. Detect or configure node capacity

**File:** `apps/api/src/bootstrap.ts:149`

**Problem:** Node capacity hardcoded to `{ vcpus: 8, memoryMb: 16384, diskGb: 100 }`.

**Fix:** Auto-detect via `os.cpus().length` and `os.totalmem()`, with env var overrides (`NODE_VCPUS`, `NODE_MEMORY_MB`, `NODE_DISK_GB`).

### 8. Make `secretStore` optional in `createApp`

**File:** `apps/api/src/bootstrap.ts:356`

**Problem:** `secretStore: secretStore!` non-null assertion crashes if no secret key is configured.

**Fix:** Make `secretStore` optional in `createApp` options type. Secret-related routes return 503 with a clear message when unavailable.

### 9. Extract `ClaimOrchestrator` from `TenantManager`

**File:** `apps/api/src/tenant-manager.ts`

**Problem:** `TenantManager` has 6 positional constructor args + 4 options, orchestrating claiming, releasing, idle monitoring, overlay management, pool replenishment, and audit logging.

**Fix:**
- Extract claim/release orchestration into a `ClaimOrchestrator` class
- Have side-effects (idle watch start/stop, pool replenish) fire via EventBus listeners instead of direct calls
- `TenantManager` becomes a thin facade that delegates to `ClaimOrchestrator` and wires EventBus handlers

### 10. Activity log retention policy

**File:** `packages/db/src/schema.ts:180-198`

**Problem:** `activity_log` table grows unbounded.

**Fix:**
- Add a `ActivityLogCleaner` that runs periodically (e.g., every hour)
- Configurable retention via `ACTIVITY_LOG_RETENTION_DAYS` env var (default 30)
- `DELETE FROM activity_log WHERE created_at < ?` with a batch limit to avoid long locks

### 13. Replace polling-based `WorkloadWatcher` with `fs.watch`

**File:** `apps/api/src/workload-watcher.ts`

**Problem:** Polls every 5 seconds. `fs.watch` provides instant detection with zero CPU overhead.

**Fix:** Use Bun's native `fs.watch` with a 500ms debounce, keeping poll as fallback for unsupported filesystems.

### 14. Add global request timeout to the API

**File:** `apps/api/src/app.ts`

**Problem:** No request timeout â€” stuck health checks or exec calls hold connections indefinitely.

**Fix:** Add Elysia `onRequest` hook with `AbortSignal.timeout(30_000)` (or configurable via `REQUEST_TIMEOUT_MS`). Claim/release routes get a longer timeout (120s).

### 15. Archive destroyed instances

**Problem:** Destroyed instances accumulate in the `instances` table.

**Fix:** Periodic cleanup job that deletes `instances` rows with `status = "destroyed"` older than N days (configurable, default 7). Instance data is already captured in activity log.

### 16. Use state machine for hibernatedâ†’destroyed transition

**File:** `apps/api/src/tenant-manager.ts:236-242`

**Problem:** `prepareNewClaim` sets hibernated instances to `status: "destroyed"` via direct `db.update()`, bypassing the state machine.

**Fix:** Use `applyInstanceTransition(db, row.instanceId, "hibernated", "destroy")` and let the FSM handle the transition properly.

### 17. Document single-node assumptions for future scaling

**Problem:** In-memory maps (`inflightClaims`, `warmingPromises`, `IdleMonitor` timers) assume single process.

**Fix:** Add `// SINGLE-NODE: requires externalization for multi-node` comments at each location. No code changes needed now â€” this is documentation for future architects.

### 18. Clarify trigger gateway vs API trigger dispatch

**Problem:** Triggers can be dispatched through both the main API and the standalone trigger-gateway binary.

**Fix:** Add a section to the README or an architecture doc explaining when to use each. The trigger-gateway is for standalone webhook receivers that don't need the full API; the API routes are for programmatic trigger management.
